### relation (0.1s)

## Relation

Think about how a grade book works. Each student is paired with a score: Alice gets 92, Bob gets 78, Carol gets 85. This pairing — who is connected to what — is exactly the idea behind a **relation**.

**Definition.** Given two sets $A$ and $B$, a *relation* $R$ from $A$ to $B$ is any subset of the Cartesian product $A \times B$. We write $a \mathrel{R} b$ (read: "$a$ is related to $b$") to mean $(a, b) \in R$. When $A = B$, we call $R$ a relation *on* $A$.

**Worked Example.** Let $A = \{1, 2, 3\}$ and define the relation $R$ on $A$ by

$$R = \{(a, b) \in A \times A \mid a \leq b\}.$$

Listing the pairs explicitly:

$$R = \{(1,1),\,(1,2),\,(1,3),\,(2,2),\,(2,3),\,(3,3)\}.$$

We have $1 \mathrel{R} 3$ because $1 \leq 3$, but $3 \not\mathrel{R} 1$ because $3 \leq 1$ is false. The relation captures *direction*: being related is not automatically mutual.

**Why this works.** Because $R \subseteq A \times A$, membership is unambiguous — an ordered pair either belongs to $R$ or it does not. There is no vagueness. Every property we care about (symmetry, transitivity, reflexivity) becomes a precise statement about which pairs are or are not in the set. For instance, $R$ is *reflexive* exactly when $(a, a) \in R$ for every $a \in A$ — a condition you can verify element by element.

**Practice Problem.** Let $A = \{2, 3, 4, 6\}$ and define $R$ by $a \mathrel{R} b$ if and only if $a$ divides $b$.

1. List all pairs in $R$.
2. Is $(4, 6) \in R$? Justify your answer.
3. Is $R$ reflexive? Check each element of $A$.

### tuple (0.3s)

## Tuple

When you look up a flight, the booking system returns not just a departure city, but a bundle of related data: origin, destination, departure time, and seat class — all locked together in a fixed order. That bundle is the everyday intuition behind a **tuple**.

**Definition.** A tuple is an ordered, finite sequence of elements. An $n$-tuple is written

$$
(a_1,\, a_2,\, \ldots,\, a_n)
$$

where $n \geq 0$ is the **arity** (or length) and each $a_i$ is called a **component**. Order and repetition both matter: $(1, 2)$ and $(2, 1)$ are distinct 2-tuples, and $(1, 1)$ is a valid 2-tuple even though both components are equal.

Two tuples are **equal** if and only if they have the same arity and every corresponding component is equal:

$$
(a_1, \ldots, a_n) = (b_1, \ldots, b_n) \iff a_i = b_i \text{ for all } i.
$$

**Worked example.** Let $A = \{0, 1\}$. List all 2-tuples with components in $A$.

$$
(0,0),\quad (0,1),\quad (1,0),\quad (1,1)
$$

There are $|A|^2 = 4$ such tuples. In general, the set of all $n$-tuples drawn from a set $A$ is the **Cartesian product** $A^n$, and $|A^n| = |A|^n$ when $A$ is finite.

**Why it works.** Because order is structural — component $i$ is always the $i$-th slot — tuples serve as a precise way to encode relations. A relation $R$ on sets $A$ and $B$ is simply a subset of the Cartesian product $A \times B$; every element of $R$ is a 2-tuple $(a, b)$ recording that $a$ and $b$ are related. Tuples turn informal "pairs of things" into first-class mathematical objects you can reason about rigorously.

**Practice.** How many 3-tuples $(x, y, z)$ satisfy $x, y, z \in \{1, 2, 3\}$ and $x + y + z = 6$? List them all.

### attribute (0.2s)

## Attribute

When a librarian catalogs a book, she records its title, author, number of pages, and genre. Each of these recorded facts is an **attribute** — a named property that can take a value for any object in a collection. Attributes are so ubiquitous that they quietly underpin databases, logic, machine learning, and formal concept analysis. Learning to reason about them precisely is one of the first steps toward structured knowledge.

**Definition.** Let $O$ be a set of *objects* and $A$ a set of *attribute names*. An *attribute* $a \in A$ is a function

$$a : O \to V_a$$

where $V_a$ is the *value domain* of $a$. When $V_a = \{0, 1\}$ (or equivalently $\{\text{false}, \text{true}\}$), the attribute is called *Boolean* (or *binary*).

**Worked example.** Consider three fruits: $O = \{\text{apple}, \text{lemon}, \text{grape}\}$ and two Boolean attributes, $\text{sweet}$ and $\text{yellow}$:

| Object | sweet | yellow |
|--------|-------|--------|
| apple  | 1 | 0 |
| lemon  | 0 | 1 |
| grape  | 1 | 0 |

Here $\text{yellow}(\text{lemon}) = 1$ and $\text{sweet}(\text{lemon}) = 0$. The pair $(O, A)$ together with an assignment table like this is called a *formal context*, written $\mathbb{K} = (O, A, I)$ where $I \subseteq O \times A$ records which objects possess which attributes.

**Why this matters.** Because an attribute is a function, two objects sharing the same attribute value form a well-defined subset of $O$. Specifically, the *extension* of attribute $a$ at value $v$ is

$$\text{ext}(a, v) = \{o \in O \mid a(o) = v\}.$$

This set-theoretic view lets us combine attributes via intersection ($\cap$) and union ($\cup$), enabling queries, classifications, and concept lattices — all grounded in ordinary set algebra.

**Practice problem.** Let $O = \{1, 2, 3, 4, 5, 6\}$ be the first six positive integers, and define $\text{even}(n) = 1$ iff $2 \mid n$, and $\text{gt3}(n) = 1$ iff $n > 3$. Write out $\text{ext}(\text{even}, 1)$ and $\text{ext}(\text{gt3}, 1)$, then compute their intersection. Which objects are both even and greater than three?

### cartesian_product (0.2s)

## Cartesian Product

A restaurant offers 3 types of bread and 4 types of filling. How many distinct sandwiches can you make? You instinctively multiply: $3 \times 4 = 12$. This counting argument is exactly the Cartesian product — a way to systematically pair every element of one set with every element of another.

**Definition.** Given sets $A$ and $B$, their *Cartesian product* is

$$A \times B = \{\,(a, b) \mid a \in A \text{ and } b \in B\,\}.$$

Each element of $A \times B$ is an *ordered pair* $(a, b)$. Order matters: $(a, b) \neq (b, a)$ in general, even when $A = B$.

**Worked Example.** Let $A = \{1, 2\}$ and $B = \{\alpha, \beta, \gamma\}$. Then

$$A \times B = \{(1,\alpha),\,(1,\beta),\,(1,\gamma),\,(2,\alpha),\,(2,\beta),\,(2,\gamma)\}.$$

This gives $|A \times B| = 2 \times 3 = 6$ ordered pairs. Notice that $B \times A$ produces the same six *combinations* but as reversed pairs $(\alpha, 1)$, $(\alpha, 2)$, etc. — a distinct set.

**Why it's true.** For each of the $|A|$ choices of $a$, there are exactly $|B|$ independent choices of $b$, so the total count is $|A| \cdot |B|$. More formally, this follows from the *multiplication principle*: independent sequential choices multiply. The Cartesian product makes this combinatorial reasoning precise by listing every outcome explicitly as a set.

This construction extends naturally: $A \times B \times C$ is the set of ordered triples, and in general $A_1 \times \cdots \times A_n$ gives $n$-tuples. The familiar coordinate plane $\mathbb{R}^2 = \mathbb{R} \times \mathbb{R}$ is the canonical infinite example.

**Practice Problem.** Let $A = \{a, b, c\}$ and $B = \{0, 1\}$. List all elements of $A \times B$ and verify that $|A \times B| = |A| \cdot |B|$. Then determine whether $A \times B = B \times A$ and justify your answer.

### domain (24.8s)

## Domain

Imagine a vending machine: you press a button, and it dispenses a snack. But not every button works — some are empty, some are out of service. The set of buttons that *actually work* is the machine's "valid input set." In mathematics, we call this the **domain**.

**Definition.** Given a function $f : A \to B$, the **domain** of $f$ is the set $A$ — the complete collection of input values for which $f$ is defined and produces an output in $B$.

When a function is given by a formula without an explicit domain, we take the **natural domain**: the largest subset of $\mathbb{R}$ (or the relevant set) for which the formula is meaningful.

**Worked Example.** Find the natural domain of

$$f(x) = \frac{\sqrt{x - 3}}{x - 7}.$$

We need two conditions to hold simultaneously:

1. The square root requires $x - 3 \geq 0$, so $x \geq 3$.
2. The denominator cannot be zero, so $x \neq 7$.

Intersecting these constraints gives

$$\text{dom}(f) = [3, 7) \cup (7, \infty).$$

**Why it's true.** The square root $\sqrt{u}$ is only real-valued when $u \geq 0$, and division by zero is undefined in any field. Since a function must assign *exactly one* output to each input, any $x$ that violates either condition cannot belong to the domain — including it would leave $f(x)$ undefined or ambiguous.

This is why specifying the domain is not a bureaucratic detail: it is part of the definition of the function itself. Two functions with the same formula but different domains are, formally, different functions.

**Practice Problem.** Determine the natural domain of

$$g(x) = \ln(x^2 - 4).$$

Express your answer in interval notation and justify each restriction.

### projection (0.1s)

## Projection

Imagine you're standing outside on a sunny afternoon. Your shadow on the ground is not you — it's a flattened version of you, cast in one direction. That shadow is a **projection**: the "shadow" of one object onto another space. This idea turns out to be one of the most powerful tools in linear algebra.

**Definition.** The *projection* of a vector $\mathbf{b}$ onto a vector $\mathbf{a}$ is the vector in the direction of $\mathbf{a}$ that is "closest" to $\mathbf{b}$. Formally,

$$\text{proj}_{\mathbf{a}} \mathbf{b} = \frac{\mathbf{b} \cdot \mathbf{a}}{\mathbf{a} \cdot \mathbf{a}} \, \mathbf{a}.$$

The scalar $\hat{c} = \dfrac{\mathbf{b} \cdot \mathbf{a}}{\mathbf{a} \cdot \mathbf{a}}$ is called the *scalar projection coefficient*; it tells you how much of $\mathbf{b}$ "lives in" the direction of $\mathbf{a}$.

**Worked example.** Let $\mathbf{a} = \begin{pmatrix}2\\1\end{pmatrix}$ and $\mathbf{b} = \begin{pmatrix}3\\4\end{pmatrix}$. Compute:

$$\mathbf{b} \cdot \mathbf{a} = 3(2) + 4(1) = 10, \qquad \mathbf{a} \cdot \mathbf{a} = 4 + 1 = 5.$$

So $\hat{c} = \tfrac{10}{5} = 2$, and

$$\text{proj}_{\mathbf{a}} \mathbf{b} = 2\begin{pmatrix}2\\1\end{pmatrix} = \begin{pmatrix}4\\2\end{pmatrix}.$$

**Why it's true.** The key insight is *orthogonality*: the error $\mathbf{e} = \mathbf{b} - \text{proj}_{\mathbf{a}}\mathbf{b}$ must be perpendicular to $\mathbf{a}$. Setting $\mathbf{e} \cdot \mathbf{a} = 0$ and solving for $\hat{c}$ yields exactly the formula above. Minimizing distance is equivalent to requiring this right angle — a geometric fact with algebraic teeth.

**Practice problem.** Let $\mathbf{a} = \begin{pmatrix}1\\2\\2\end{pmatrix}$ and $\mathbf{b} = \begin{pmatrix}6\\0\\3\end{pmatrix}$. Find $\text{proj}_{\mathbf{a}}\mathbf{b}$ and verify that $\mathbf{b} - \text{proj}_{\mathbf{a}}\mathbf{b}$ is orthogonal to $\mathbf{a}$.

### selection (0.1s)

## Selection

You walk into an ice cream shop with ten flavors and want to choose three scoops for your bowl. You don't care about order — chocolate-vanilla-strawberry is the same as strawberry-chocolate-vanilla. How many distinct bowls are possible? This everyday puzzle is the heart of **selection**.

**Definition.** A *selection* (or *combination*) of $r$ items from a set of $n$ distinct items, where order does not matter, is counted by the *binomial coefficient*:

$$\binom{n}{r} = \frac{n!}{r!\,(n-r)!}$$

Read $\binom{n}{r}$ as "$n$ choose $r$." The formula counts how many ways to pick an unordered subset of size $r$ from $n$ objects.

**Worked Example.** Back to the ice cream shop with $n = 10$ flavors and $r = 3$ scoops:

$$\binom{10}{3} = \frac{10!}{3!\cdot 7!} = \frac{10 \times 9 \times 8}{3 \times 2 \times 1} = \frac{720}{6} = 120$$

There are exactly 120 distinct bowls. Notice the numerator counts ordered triples ($10 \times 9 \times 8 = 720$), and dividing by $3! = 6$ removes the overcounting caused by the $3! = 6$ orderings of any single set of three flavors.

**Why it's true.** Start by counting *ordered* selections: the first scoop has $n$ choices, the second $n-1$, the third $n-2$, giving $n(n-1)(n-2) = \dfrac{n!}{(n-r)!}$ ordered lists. Every unordered set of $r$ items appears exactly $r!$ times among those lists (once per permutation of the same items). Dividing removes all duplicates:

$$\binom{n}{r} = \frac{n!}{r!\,(n-r)!}$$

**Practice Problem.** A committee of 4 people is to be chosen from 9 candidates. How many committees are possible? Compute $\binom{9}{4}$ and verify your answer is 126.

### index (15.7s)

## Index

When you look up a word in a dictionary, you don't read every page — you jump straight to the right letter. An **index** in mathematics serves the same purpose: it is a label that pinpoints a specific element inside a collection without ambiguity.

**Definition.** Given a sequence or collection of objects $a_1, a_2, a_3, \ldots, a_n$, the **index** is the subscript $i$ in $a_i$ that identifies which element you are referring to. More precisely, an index is an element of an **index set** $I$, and an indexed family is a function $f: I \to S$ where $S$ is some set of values. We write the image of $i$ as $a_i$ rather than $f(i)$ to emphasize the positional role of $i$.

**Worked example.** Consider the sequence of squares $a_i = i^2$ for $i \in \{1, 2, 3, 4, 5\}$:

$$a_1 = 1,\quad a_2 = 4,\quad a_3 = 9,\quad a_4 = 16,\quad a_5 = 25.$$

The index $i = 3$ selects exactly $a_3 = 9$. Summation notation packages all five values cleanly:

$$\sum_{i=1}^{5} a_i = \sum_{i=1}^{5} i^2 = 1 + 4 + 9 + 16 + 25 = 55.$$

Here the index $i$ is a **bound variable** — it exists only inside the sum and carries no meaning outside it.

**Why it works.** An index set imposes an ordering (or at least a labeling) on what would otherwise be an unstructured collection. Because each label maps to exactly one element, retrieval is unambiguous. This is precisely the definition of a function, so indexed families inherit all the machinery of function theory — domains, ranges, injectivity — for free.

**Practice problem.** Let $b_k = 2k - 1$ for $k \in \{1, 2, \ldots, n\}$.

1. Write out $b_1, b_2, b_3, b_4$.
2. Compute $\displaystyle\sum_{k=1}^{n} b_k$ and simplify. What familiar formula do you recognize?

### join (0.2s)

## Join

When you need to schedule a meeting that works for two people, you naturally look for the *earliest* time slot that fits *both* calendars. That intuition—finding the smallest thing that "covers" two given things—is exactly what a **join** formalizes in mathematics.

**Definition.** Let $(P, \leq)$ be a partially ordered set. The *join* of two elements $a, b \in P$, written $a \vee b$, is the *least upper bound* of $\{a, b\}$: the element $u \in P$ such that

$$a \leq u, \quad b \leq u, \quad \text{and} \quad \forall v \in P,\ (a \leq v \text{ and } b \leq v) \implies u \leq v.$$

A partially ordered set in which every pair of elements has a join (and a *meet*) is called a **lattice**.

**Worked example.** Take $P = \mathcal{P}(\{1,2,3\})$, the power set of $\{1,2,3\}$, ordered by set inclusion $\subseteq$. Let $A = \{1,2\}$ and $B = \{2,3\}$. Which sets contain both $A$ and $B$ as subsets? Any superset of $A \cup B = \{1,2,3\}$, but the *smallest* one is $\{1,2,3\}$ itself. So

$$A \vee B = \{1,2,3\}.$$

Here join coincides with union—a pattern that holds in every power-set lattice.

**Why it's true.** Uniqueness of the join follows directly from antisymmetry: if $u$ and $u'$ both satisfy the definition, then $u \leq u'$ and $u' \leq u$, forcing $u = u'$. Existence, however, is not guaranteed in every poset—it must be assumed or proved case by case.

**Practice problem.** Consider the divisibility order on $\mathbb{Z}_{>0}$: say $a \leq b$ when $a \mid b$. Compute $6 \vee 10$. What familiar number-theoretic operation does join correspond to here?

### transaction (16.8s)

## Transaction

Every time you transfer money between bank accounts, you expect two things to happen together: the source account decreases and the destination account increases. If the system crashes halfway through, you don't want one to happen without the other. This everyday expectation is formalized by the concept of a **transaction**.

**Definition.** A *transaction* is a finite sequence of operations $T = \langle o_1, o_2, \ldots, o_n \rangle$ executed on a shared data store, satisfying the **ACID** properties:

- **Atomicity** — either all operations commit, or none do.
- **Consistency** — $T$ maps a valid database state to another valid database state.
- **Isolation** — concurrent transactions behave as if they were executed serially.
- **Durability** — once committed, the effects of $T$ persist despite failures.

**Worked example.** Let account balances be $A = 500$ and $B = 200$. Consider a transfer of $\$100$:

$$T = \langle \text{read}(A),\; A \gets A - 100,\; \text{write}(A),\; \text{read}(B),\; B \gets B + 100,\; \text{write}(B) \rangle$$

If the system fails after `write(A)` but before `write(B)`, the database would show $A = 400,\, B = 200$ — $\$100$ has vanished. Atomicity prevents this: a *rollback* restores $A = 500$, leaving the state as if $T$ never ran. After a successful commit we have $A = 400,\, B = 300$, and the invariant $A + B = 700$ is preserved (consistency).

**Why it works.** Systems enforce atomicity via a *write-ahead log* (WAL): every intended change is recorded to durable storage before it is applied. On recovery the engine replays committed log entries and undoes incomplete ones, guaranteeing the all-or-nothing semantic regardless of when failure occurs.

**Practice problem.** Suppose three concurrent transactions each read a shared counter $C = 10$ and each write $C \gets C + 1$. Without isolation, what values could $C$ hold after all three commit? How many distinct outcomes are possible, and which single outcome does strict serializability guarantee?

### web_application_backend (14.4s)

## Web Application Backend

Every time you search for a product on an e-commerce site, something invisible handles your request: it looks up your account, queries a database of millions of items, applies filters, and returns a personalised list — all within milliseconds. That invisible layer is the **backend**.

**Definition.** A web application backend is the server-side component of a web system responsible for business logic, data persistence, and communication between clients and databases. Formally, we can model it as a function

$$B: (R, S) \rightarrow (R', S')$$

where $R$ is an incoming HTTP request, $S$ is the current application state (database contents, session data), $R'$ is the HTTP response returned to the client, and $S'$ is the updated state after the request is processed.

**Worked Example.** Consider a simple login endpoint. A client sends a POST request containing credentials $\{u, p\}$. The backend:

1. Receives $R = \texttt{POST /login}$ with body $\{u, p\}$.
2. Queries the database: $\texttt{SELECT hash FROM users WHERE username} = u$.
3. Verifies: $\texttt{verify}(p,\, \texttt{hash}) \in \{\text{true}, \text{false}\}$.
4. If true, creates a session token $\tau$ and returns $R' = 200\,\texttt{OK}$ with $\{\tau\}$; otherwise returns $R' = 401\,\texttt{Unauthorized}$.
5. State transitions: $S' = S \cup \{(\tau, u)\}$ if login succeeds, else $S' = S$.

**Why it works.** The mapping $B$ must be **deterministic for identical inputs** (same credentials always produce the same verification outcome) yet **stateful** (the token $\tau$ persists across subsequent requests). This separation of pure logic from mutable state is what makes backends testable and scalable independently of the frontend.

**Practice Problem.** Design the state-transition function $B$ for a `/logout` endpoint. Define what $R$, $S$, $R'$, and $S'$ should be, and explain what invariant must hold about $S'$ after a successful logout.
