## 0. High-level Description

This script implements a minimal passthrough (proxy) pattern in SPL: it defines a single `CREATE FUNCTION` named `answer` whose body is a bare `{prompt}` slot — no formatting, no instructions, no sentinel tokens — making it a transparent conduit that forwards the caller's text verbatim to whatever model is configured at runtime. The workflow is declared with the `PROMPT` keyword as `ollama_proxy` rather than a full `WORKFLOW` block, reflecting its role as a lightweight, single-step invocation rather than a multi-stage orchestration. A `system_role` directive injects a fixed assistant persona ("You are a helpful, knowledgeable assistant.") before the user content, which arrives via `context.prompt`. The single `GENERATE answer(prompt)` call drives the LLM and implicitly returns its output. Because the adapter and model are supplied as CLI flags (`--adapter`, `--model`) rather than `INPUT` variables inside the script, this workflow is model-agnostic by design: the same `.spl` file can target Ollama (gemma3, llama3.2, mistral) or Anthropic without any script modification. There is no `WHILE` loop, no `EVALUATE` branch, no `EXCEPTION` handling, no `CALL` side-effects, and no `LOGGING` — the script is deliberately minimal, optimised for the single responsibility of proxying a prompt to an LLM and returning its answer.

---

## 1. Purpose

Forward an arbitrary user-supplied prompt to any configured LLM adapter/model and return the model's raw response, acting as a universal LLM query proxy.

---

## 2. Inputs

| Parameter | Default | Description |
|-----------|---------|-------------|
| `prompt` | *(required)* | The user prompt or question to send to the model. Passed via CLI as a named argument (`prompt="..."`). |

> **Note:** `--adapter` and `--model` are runner-level flags consumed by the `spl3` CLI, not `INPUT` variables declared inside the script.

---

## 3. Process

1. The `spl3` runtime resolves the target adapter and model from the CLI flags `--adapter` and `--model`.
2. The `PROMPT ollama_proxy` block is entered; `context.prompt` is bound to the value of the `prompt` CLI argument.
3. A system persona is attached to the request via `system_role('You are a helpful, knowledgeable assistant.')`.
4. `GENERATE answer(prompt)` is called: the `answer` function substitutes `{prompt}` with the user's text and dispatches the completed prompt to the configured LLM.
5. The model's response is captured and returned as the workflow's output.

---

## 4. Error Handling

*(None declared.)* The script contains no `EXCEPTION` blocks. Any runtime errors (model unavailable, malformed response, budget exceeded, context overflow, etc.) propagate unhandled to the `spl3` CLI, which surfaces them to the caller.

---

## 5. Output

The raw text response generated by the LLM for the given prompt. No explicit `RETURN` statement with metadata fields (status, iteration count, etc.) is present; output is the direct result of the single `GENERATE` call.
