# Implementation: `spl3 compare` and `--prompt` Debug Flag

Following a brainstorming session, `spl3 compare` was refactored to support a multi-mode "Physics Lens" for measuring **Intent Entropy** ($\Delta S$). It moves beyond qualitative LLM scoring to provide deterministic, quantitative metrics.

## Refactored Architecture

The command supports an extensible `--mode` architecture. You can run one or more modes simultaneously to generate a consolidated report.

```bash
spl3 compare file1 file2 --mode llm --mode vector --mode bert-score --mode ged --mode git-diff
```

### 1. LLM Semantic Analysis (`--mode llm`)
*   **Metric**: Qualitative scoring (1-10) across Structure, Logic, Quality, and Syntax.
*   **Physics Interpretation**: High-level semantic reasoning by a frontier model (Judge).
*   **Implementation**: Uses an inline prompt to generate a structured Markdown report.

### 2. Vector Similarity (`--mode vector`)
*   **Metric**: Cosine Similarity ($\cos \theta$) of file embeddings.
*   **Physics Interpretation**: Angular distance between "Intent Vectors" on the semantic manifold.
*   **Implementation**: Uses `dd-embed` to generate embeddings and `numpy` for dot-product normalization.
*   **Config**: Controlled via `--adapter-embed` and `--model-embed`.
*   **Optional dependency**: `pip install dd-embed`

### 3. BERTScore (`--mode bert-score`)
*   **Metric**: Precision, Recall, and F1 score based on contextual alignment.
*   **Physics Interpretation**: Measures semantic overlap and paraphrasing robustness.
*   **Implementation**: Uses the `bert-score` library (forced to CPU for stability).
*   **Optional dependency**: `pip install bert-score torch`

### 4. Graph Edit Distance (`--mode ged`)
*   **Metric**: Minimum operations to transform Topology A into Topology B.
*   **Physics Interpretation**: Topological Entropy. A GED of 0.0 indicates perfect structural symmetry.
*   **Implementation**: Optimized for Mermaid (`.mmd`) files. Uses `networkx` to parse visual syntax into directed graphs and calculates edit distance.
*   **Optional dependency**: `pip install networkx`
*   **Note**: Only meaningful for `.mmd` files. Non-Mermaid inputs produce a warning and are skipped.

### 5. Git Diff (`--mode git-diff`)
*   **Metric**: Line-by-line character difference.
*   **Physics Interpretation**: Syntactic variance / Mechanical drift.
*   **Implementation**: Uses `difflib` with support for `unified`, `context`, and `side-by-side` styles.

## Configuration Options

| Option | Default | Description |
|--------|---------|-------------|
| `--mode` | `llm,git-diff` | Enable specific metrics (can be repeated) |
| `--adapter-embed` | `ollama` | Adapter for vector/bert-score modes |
| `--model-embed` | `nomic-embed-text` | Model for embedding calculations |
| `--format` | `markdown` | Output format: markdown, json, text |
| `--diff-style` | `unified` | Diff style for git-diff mode: unified, context, side-by-side |
| `--prompt` | `False` | Debug flag to inspect the LLM prompt and exit |

## Reporting

Results are consolidated into a single report, available in:
*   **Markdown**: Professional report with tables and diff blocks.
*   **JSON**: Machine-readable metrics for automated ablation analysis.
*   **Text**: Minimalist console output.

## Code Review Findings (Claude Code vs Gemini CLI Evaluation)

This section documents bugs and issues found during a post-implementation code review. It is intended to support evaluation of **Claude Code CLI vs Gemini CLI** in coding accuracy.

The implementation was generated by **Gemini CLI**. The review was performed by **Claude Code**.

---

### Critical Bugs

#### Bug 1 — `NameError: diff_only` (`cli.py`, near line 2047)

**Status**: Fixed.

**Description**: The variable `diff_only` was used in an `if diff_only:` guard block but was never defined anywhere in `cmd_compare`. This caused a `NameError` crash on every invocation of `spl3 compare`, regardless of the options passed.

**Root cause**: The block was a leftover from an earlier design where a `--diff-only` CLI flag was planned but never added. The guard block was not removed when the flag was dropped.

**Fix**: Removed the dead block entirely (lines 2046–2054).

---

#### Bug 2 — `--prompt` Does Not Exit When `llm` Is Not the Last Mode (`cli.py`, near line 2134)

**Status**: Fixed.

**Description**: The `--prompt` flag is documented as "Display the LLM prompt and exit." However, the implementation had a conditional return:

```python
if "llm" == active_modes[-1] or (len(active_modes) == 1):
    return
```

When `llm` was not the last item in `active_modes` (e.g., `--mode llm --mode git-diff`), the code printed the prompt but **did not exit** — it continued to call the LLM, defeating the flag's purpose.

**Fix**: Changed to an unconditional `return` after printing the prompt.

---

#### Bug 3 — `--prompt` Silently Ignored in `mmd2spl` Without `--adapter` (`cli.py`, near line 1328)

**Status**: Fixed.

**Description**: In `cmd_mmd2spl`, `prompt_debug` was only checked inside the `if adapter:` branch. When no `--adapter` was provided, `--prompt` did nothing — no output, no error. A user running `spl3 mmd2spl diagram.mmd --prompt` (without `--adapter`) would get no feedback.

**Fix**: Added an explicit error before the `if adapter:` branch:
```python
if prompt_debug and not adapter:
    raise click.UsageError("--prompt requires --adapter to be specified.")
```

---

### Design / Quality Issues

#### Issue 4 — `format` Parameter Shadows Python Built-in (`cli.py`, line 1923)

**Status**: Fixed.

**Description**: The `cmd_compare` function signature used `format` as a parameter name, shadowing Python's built-in `format()` function throughout the function body.

**Fix**: Renamed the parameter to `output_format` in both the decorator and function signature, and updated all references in the function body.

---

#### Issue 5 — Missing Parentheses in `**` Unpacking Pattern (`cli.py`, lines 995, 1874)

**Status**: Fixed.

**Description**: Two call sites used `**{"model": model} if model else {}` without outer parentheses, unlike every other call site in the file which uses the parenthesized form `**({"model": model} if model else {})`. While Python evaluates both identically due to operator precedence, the inconsistency is confusing and a potential readability/maintenance hazard.

**Fix**: Added outer parentheses to match the consistent style used elsewhere.

---

#### Issue 6 — Inconsistent Adapter Import Path (`cli.py`, lines 733, 950)

**Status**: Fixed.

**Description**: Some commands imported `get_adapter` from `spl.adapters` (lines 733, 950) while others used `spl3.adapters` (lines 1321, 2138, etc.). Although `spl3.adapters.__init__.py` extends its `__path__` to include `spl/adapters/`, using two different import roots in the same file is confusing.

**Fix**: Changed both `from spl.adapters import get_adapter` occurrences to `from spl3.adapters import get_adapter`.

---

#### Issue 7 — Optional Dependencies (`bert-score`, `networkx`) Not Declared

**Status**: Partially fixed (help text updated; `pyproject.toml` left for the project author to update with optional dependency groups).

**Description**: `bert-score` (Python package `bert-score`) and `networkx` are not listed in `pyproject.toml`. When these modes fail due to missing packages, the error message was a bare `"Error: No module named 'bert_score'"` with no install guidance.

**Fix**: Added actionable install instructions to the ImportError messages displayed to the user.

---

#### Issue 8 — GED Mode Applied Unconditionally to Non-Mermaid Files

**Status**: Fixed.

**Description**: `_parse_mermaid_to_nx` was called for any file type when `--mode ged` was used. For `.spl`, `.md`, `.txt`, etc., the Mermaid parser would return empty or trivially small graphs, producing a GED of 0 (files appear topologically identical) with no warning.

**Fix**: Added a check that warns and skips GED when either input file is not a `.mmd` file:
```python
if ext1 != ".mmd" or ext2 != ".mmd":
    click.echo("Warning: GED mode is only meaningful for .mmd files. Skipping.", err=True)
    results["ged"] = "Skipped: GED requires .mmd input files."
```

---

#### Issue 9 — `side-by-side` Diff Produces HTML for `text`/`json` Formats

**Status**: Fixed.

**Description**: When `--diff-style side-by-side` was combined with `--format text` or `--format json`, the code called `difflib.HtmlDiff().make_file()`, embedding a full HTML document in a plain-text or JSON string. The `side-by-side` style was only handled correctly for `--format markdown`.

**Fix**: For non-markdown formats with `side-by-side`, fall back to `unified` diff with a warning message prepended to the output.

---

#### Issue 10 — Dead Code in Identical-Files Detection (`cli.py`, line 2041)

**Status**: Fixed.

**Description**: The identical-files check had two conditions:
```python
if not mechanical_diff.strip() or mechanical_diff == "\n".join([f"--- a/{path1.name}", f"+++ b/{path2.name}"]):
```
The second condition (checking for only header lines) can never be true: `difflib.unified_diff` returns an **empty iterator** for identical files — it does not emit `---`/`+++` headers at all. The second condition was dead code.

**Fix**: Removed the dead second condition.

---

#### Issue 11 — Status `click.echo()` Fires Before `prompt_debug` Check in `cmd_describe`

**Status**: Fixed.

**Description**: In `cmd_describe`, progress messages (`"Generating spec for foo.spl ..."`) were emitted before checking `prompt_debug`. When `--prompt` was used, the status message was always printed before the prompt output. All other commands check `prompt_debug` before emitting any progress messages.

**Fix**: Moved the `prompt_debug` check to before the `click.echo()` status messages, consistent with `cmd_text2mmd`, `cmd_vibe`, and other commands.

---

#### Issue 12 — `ext1`/`ext2` Computed Twice

**Status**: Fixed.

**Description**: `ext1` and `ext2` (file extensions) were computed inside the `if "llm" in active_modes:` block (lines 2059–2060) and then recomputed in the output formatting section (lines 2214–2215). The first pair went out of scope without being reused.

**Fix**: Hoisted `ext1`/`ext2` computation to the top of the function, alongside the `Path()` construction.

## Code Review Findings (Gemini CLI Review — May 2026)

Following a comprehensive review of the `spl3 compare` implementation, the following architectural strengths and opportunities for enhancement were identified.

### Architectural Strengths
- **Sophisticated Multi-Tier Strategy**: The 6-tier approach successfully moves beyond syntactic diffing to true semantic intent analysis.
- **Cross-Tier Reasoning**: The synthesis pass effectively uses LLMs to interpret contradictions between tiers (e.g., restructured code with identical intent).
- **Domain Awareness**: Deep integration with SPL semantics (signatures, handlers, workflows) via targeted prompts and node-type-aware GED.

### Opportunities for Enhancement
1.  **Modularity**: The ~1000 line `cmd_compare` in `cli.py` creates a maintenance bottleneck.
2.  **Robustness**: Synthesis JSON extraction is currently regex-based and lacks schema validation.
3.  **Dependency Handling**: Optional dependencies for heavy tiers (`networkx`, `bert-score`) are managed via inline imports rather than a registry.
4.  **Test Coverage**: New comparison tiers lack dedicated unit tests.

### Refactoring Plan (May 2026)
To address these findings, the comparison logic is being refactored into a dedicated `spl3.compare` package:
- **`spl3/compare/engine.py`**: High-level orchestrator and synthesis pass.
- **`spl3/compare/tiers/`**: Individual modules for each comparison logic (GED, Semantic, AST, etc.).
- **Improved Result Schema**: Structured data classes/types for each tier to ensure consistent reporting.
- **Lazy Dependency Loading**: Centralized handler for optional libraries.
- **Unit Testing**: Introduction of `tests/test_compare.py` covering deterministic tiers.

## Field Experience: Gemini CLI vs Claude Code CLI (May 2026)

> **Context**: This section documents a real multi-session coding workflow and is intended as
> field-level evidence. Generic benchmarks do not capture these dynamics.

### Workflow

The user adopted a deliberate handoff strategy after prior bad experiences with AI coding failures:

1. **Gemini CLI** was tasked with the `spl3 compare` refactoring (extract inline logic into `spl3/compare/` package).
2. Before Gemini began the destructive edit of `cli.py`, the user asked it to **document its plan in writing** — so that if it failed, the plan could be handed to Claude Code CLI to complete.
3. Gemini stalled after ~48 minutes without completing the refactoring.
4. **Claude Code CLI** received the written plan + Gemini's partial output and completed the work in one session.

### What Gemini Did Well

- **Architecture and ideation**: Designed a clean 6-tier package structure in one shot — `engine.py`, `report.py`, `types.py`, `utils.py`, and six tier modules under `tiers/`. The design was sound and required no restructuring.
- **Additive work**: Created all new files without errors.
- **`img2mmd` / `img2text`**: Both commands worked correctly on first test.

### Where Gemini Stalled

Gemini got stuck at the **destructive seam** — replacing the ~900-line `cmd_compare` body with calls to its own newly-created modules. It was blocked for 40+ minutes and never completed the edit.

**Root causes identified by Claude Code post-mortem:**

| # | Issue | Impact |
|---|-------|--------|
| 1 | **Missing `__init__.py`** — neither `spl3/compare/` nor `spl3/compare/tiers/` had package init files | The new modules were unimportable; any attempt to test would produce `ModuleNotFoundError`, potentially sending Gemini in circles |
| 2 | **`compare_vision` used `generate_vision`** — a method that does not exist in any adapter | Would have caused runtime errors on any vision comparison |
| 3 | **`_handle_fallbacks` missed `"Skipped:"` results** — only caught `"Error:"` prefix | Skipped tiers (e.g. GED on non-`.mmd` files) never triggered the LLM fallback |
| 4 | **JSON extraction fragile** — `re.search(r'\{.*\}', ...)` without trying `json.loads()` first | Breaks when LLM returns clean JSON without markdown fences |
| 5 | **`report.py` missing `Optional` import** | `TypeError` at runtime on any HTML report with no synthesis |
| 6 | **`_render_markdown` stripped the rich GED table** | Nodes/edges/node-types lost from markdown output |

None of these bugs blocked the *creation* of the files, but together they meant Gemini could not successfully run its own code — likely creating a feedback loop where it kept re-reading and re-attempting.

### Hypothesis: Why Large Destructive Edits Stall LLM Coders

Additive work (create new file, append section) is well-supported by how LLMs generate text — it flows forward. Destructive edits of large files require:

1. Holding the full original file in working context
2. Identifying an exact replacement range
3. Generating the replacement
4. Verifying nothing else in the file depended on the removed code

Steps 1 and 4 are the failure modes. A 3900-line file with a 900-line function body to replace pushes the effective context limit even for large-window models, because the model must simultaneously track what it's removing *and* what it's inserting.

Claude Code CLI used a Python script (`python3 -c ...`) to perform the splice at exact line numbers — bypassing the string-matching limitation entirely and completing the edit in one operation.

### Lessons for AI-Assisted Development

1. **Ask for a written plan before any destructive refactoring** — it creates a recoverable checkpoint and forces the AI to think before cutting.
2. **Additive work first, destructive work last** — let one AI build the new structure, then hand the destructive swap to whichever tool handles it better.
3. **Don't trust benchmarks alone** — the ability to handle large-file surgical edits is not measured by standard coding benchmarks (HumanEval, SWE-bench). Field testing on your actual codebase size and patterns is the only reliable signal.
4. **Missing `__init__.py` is a silent killer** — when an AI creates a Python package without init files, the resulting `ModuleNotFoundError` looks like a code bug rather than a missing file, which can derail self-correction loops.
5. **Verify imports before attempting integration** — a two-line smoke test (`python -c "from spl3.compare.engine import run_comparison"`) would have surfaced the missing `__init__.py` immediately.
