# Text2SPL — Recipe-Driven Development Guide

`spl3 text2spl` compiles a natural-language description into a `.spl` workflow.
Because SPL is a new language with no pre-existing LLM training data, the quality
of generated code depends entirely on the examples we give it at compile time.
The pipeline below shows how we bootstrap that knowledge from our own cookbook.

---

## The Core Idea

```
cookbook/*.spl
    └─► spl3 describe        →  *-spec.md  (Section 0 = rich SPL-aware description)
            └─► code-rag seed →  vector store  (semantic index of examples)
                    └─► spl3 text2spl  →  retrieves nearest examples as few-shot context
                                └─► new .spl workflow  (good baseline, iterate from here)
```

No fine-tuning needed. The RAG store gives `text2spl` exactly the few-shot
examples it needs to understand SPL syntax, patterns, and idioms.

---

## The `spl3 describe` Command

`spl3 describe` reads a `.spl` file and produces a `*-spec.md` with five sections.
The critical one for code generation is **Section 0: High-level Description** —
4–6 sentences of prose written in SPL vocabulary (`WORKFLOW`, `CREATE FUNCTION`,
`WHILE`, `EVALUATE`, `EXCEPTION`, etc.) that capture the pattern, control flow,
and key conventions of the script.

```bash
spl3 describe cookbook/05_self_refine/self_refine.spl --adapter claude_cli
# Writes: cookbook/05_self_refine/self_refine-spec.md
```

Section 0 from this example reads like:

> *"This WORKFLOW implements the self-refine pattern using three CREATE FUNCTION
> helpers — draft, critique, and refine. The critic function uses the [APPROVED]
> sentinel token to signal acceptance. A WHILE loop guarded by @max_iterations
> drives the cycle; an EVALUATE branch on the feedback triggers either an early
> RETURN or a further GENERATE refine() call. CALL write_file() side-effects log
> every intermediate draft and feedback to @log_dir. EXCEPTION handlers for
> MaxIterationsReached and BudgetExceeded ensure graceful termination."*

That prose is what gets indexed into the RAG store and retrieved when a user asks
`text2spl` for "self-refine something".

## Vector Store

The Code-RAG store is backed by **ChromaDB** (via `dd_vectordb.ChromaVectorDB`)
with embeddings generated by `sentence-transformers/all-MiniLM-L6-v2` (via `dd_embed`).
ChromaDB persists to a single SQLite file — no server process required.

### Default location

```
<project-root>/.spl/code_rag/chroma.sqlite3
```

The path is controlled by the `--storage-dir` option (default: `.spl/code_rag`).
The collection name inside ChromaDB is always `spl_code_rag`.

### Backup

Copy the entire directory — one file is all there is:

```bash
cp -r .spl/code_rag/ ~/backups/spl_code_rag_$(date +%Y%m%d)/
```

### Migration to another machine

```bash
# On the source machine
cp -r .spl/code_rag/ /tmp/spl_code_rag_backup/

# Transfer (adjust to your method)
rsync -av /tmp/spl_code_rag_backup/ user@target:/path/to/project/.spl/code_rag/

# On the target machine — verify
spl3 code-rag stats
```

### Clean slate / rebuild from scratch

Delete the store directory and re-run the Initial Load:

```bash
cd $HOME/projects/digital-duck/SPL.py
rm -rf .spl/code_rag/
spl3 code-rag describe-all cookbook/ --adapter claude_cli \
  --catalog cookbook/cookbook_catalog.json
spl3 code-rag seed cookbook/ --from-specs
spl3 code-rag stats


# test
spl3 text2spl "self-refine a product description with writer and critic" \
    --mode workflow --adapter claude_cli \
    -o cookbook/05_self_refine/self_refine-product_gen.spl 
```

> The `-spec.md` files in the cookbook are the source of truth — as long as those
> exist, the vector store can always be rebuilt without re-calling the LLM.

## Step 2 — Code-RAG Store Pipeline

The vector store is managed as an ETL pipeline with two distinct load patterns,
borrowed from data engineering practice.

### Initial Load

Run once to build the store from scratch. This covers all active recipes in the
catalog that do not yet have a spec file.

```bash
# 1. Extract + Transform: generate spec files for all active catalog recipes
spl3 code-rag describe-all cookbook/ \
  --adapter claude_cli \
  --catalog cookbook/cookbook_catalog.json

# 2. Load: index every spec's Section 0 into the vector store
spl3 code-rag seed cookbook/ --from-specs

# 3. Verify
spl3 code-rag stats
# Code-RAG store: .spl/code_rag
#   Indexed pairs: 47
```

`describe-all` notes:
- Filters to `is_active: true` and `approval_status` not in `disabled/rejected`
- `--skip-existing` is on by default — safe to re-run without re-processing done work
- Writes each `<name>-spec.md` alongside its `.spl` file (use `--spec-dir` to centralise)
- 2 image recipes (#51, #57) are skipped due to a catalog path quirk — harmless warnings

---

### Incremental Load

Run whenever a recipe changes or a new one is added. Mirrors the ETL
"upsert on change" pattern — only touch what has actually changed.

#### New recipe added to the cookbook

```bash
# 1. Write your new .spl file and add it to cookbook_catalog.json

# 2. Generate its spec (describe-all --skip-existing skips everything already done)
spl3 code-rag describe-all cookbook/ \
  --adapter claude_cli \
  --catalog cookbook/cookbook_catalog.json

# 3. Re-seed — new spec is picked up, existing entries are unchanged
spl3 code-rag seed cookbook/ --from-specs
spl3 code-rag stats
```

#### Existing recipe updated (logic or prompt changed)

When a `.spl` file changes its behaviour, its spec is stale and must be
regenerated before re-seeding so the RAG store reflects the new truth.

```bash
# 1. Re-describe the changed recipe, overwriting the stale spec
spl3 describe cookbook/05_self_refine/self_refine.spl --adapter claude_cli

# 2. Re-seed the full store (picks up the updated Section 0)
spl3 code-rag seed cookbook/ --from-specs
spl3 code-rag stats
```

> **Note:** `spl3 code-rag seed --from-specs` performs a full re-index of all
> spec files currently present on disk. It is idempotent — running it multiple
> times is safe. If you need a clean slate (e.g. to remove a deleted recipe),
> delete `.spl/code_rag/` and re-run the full Initial Load sequence.

## Step 3 — Generate a New Workflow with text2spl

With the RAG store primed, `text2spl` automatically retrieves the most relevant
cookbook examples and injects them as few-shot context into the LLM prompt.

```bash
spl3 text2spl "self-refine a product description with a writer and critic model" \
  --mode workflow \
  --adapter claude_cli \
  --output my_workflow.spl
```

When the query matches a known pattern (e.g. "self-refine"), the Section 0 of
`self_refine-spec.md` is retrieved — it mentions `CREATE FUNCTION draft/critique/refine`,
the `[APPROVED]` sentinel, the `WHILE` loop, `EVALUATE` branch, and `EXCEPTION`
types — giving the LLM a precise SPL template to work from.

**This is never one-shot magic.** The output is a solid baseline. Iterate on it:

1. Run `spl3 validate my_workflow.spl` to catch syntax errors
2. Run `spl3 explain my_workflow.spl` to review the execution plan
3. Run `spl3 run my_workflow.spl --adapter claude_cli` to test it
4. Edit and refine — use `spl3 describe my_workflow.spl` to re-document as it evolves

---

## Step 4 — Close the Loop

Every new workflow you write is a potential cookbook recipe. Once it runs cleanly,
promote it and feed it back into the RAG store via the Incremental Load flow above.
The store grows with the cookbook — each new recipe makes `text2spl` smarter for
the next generation task.

```
new .spl  →  catalog entry  →  describe  →  spec  →  code-rag seed  →  richer RAG
```

---

## Querying the RAG Store

Test what examples would be retrieved for a given description:

```bash
spl3 code-rag query "iterative refinement with critic model" --top-k 3
```

Useful for understanding why `text2spl` generated a particular structure, or for
debugging retrieval quality before running a full compilation.

## Summary

```bash
# ── Initial Load (run once) ────────────────────────────────────────────────
spl3 code-rag describe-all cookbook/ --adapter claude_cli \
  --catalog cookbook/cookbook_catalog.json
spl3 code-rag seed cookbook/ --from-specs
spl3 code-rag stats

# ── Incremental Load (new recipe) ─────────────────────────────────────────
spl3 code-rag describe-all cookbook/ --adapter claude_cli \
  --catalog cookbook/cookbook_catalog.json   # --skip-existing handles the rest
spl3 code-rag seed cookbook/ --from-specs

# ── Incremental Load (changed recipe) ─────────────────────────────────────
spl3 describe cookbook/<nn_name>/<name>.spl --adapter claude_cli
spl3 code-rag seed cookbook/ --from-specs

# ── Daily use ──────────────────────────────────────────────────────────────
spl3 text2spl "<your description>" --mode workflow --adapter claude_cli -o out.spl
spl3 validate out.spl
spl3 run out.spl --adapter claude_cli
```

The recipe-based development flow is self-reinforcing: every `.spl` you write
enriches the RAG store, which improves the next `text2spl` generation, which
produces better baselines to iterate from.
